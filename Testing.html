Testing

Due to the size and scale of ComCare project, there will be a need for multiple stages of testing to ensure the project is successful. There are many independent elements to our system that will all need to work in sync to ensure a smooth and efficient experience for all:

  • ComCare infrastructure and data management.
  • Connection between supermarkets and ComCare to collect inventory data,
  • ComCare inventory analysis front end.
  • ComCare restriction analysis front end.
  • Retail checkout system.
  • User Application and web interfaces.
    • Family structure, special requirements, location and so forth to be communicated back to ComCare data centres.
    • Restrictions when necessary to be sent and enforced (through the retail checkout system) through the ComCare app and card

Overall all systems will need to communicate with an entity relationship as follows:

Figure 8. ERD for the ComCare Database Management System

As this is a very involved project trying to test every section at once will open ourselves up to complete potential redesigns if we have issues. The strategy we have decided to use is, we will be breaking all major components up ensuring that every component off the project is independently tested for issues. This allows the team to address issues in each segment before testing the next. Overall decreasing the impact of the issue on the system and time it will take to resolve

Our initial testing block will consist of the connection between ComCare infrastructure and the supermarket inventory systems. Primary ensuring that all required data can be sent from the supermarket to ComCare in a timely, secure and effective method. There will be heavy involvement between ComCare and the supermarket chains for this process. To ensure the time and process is effective, we will be assigning a development and UAT testing team to this process. Reaching out to the relevant supermarket chains to test and develop.

Once the communication layer between supermarkets and ComCare has been tested and we can ensure the communication is robust the next phase of testing will be that of the ComCare inventory analysis front end. This is where employees using ComCare will be able to monitor ongoing disaster areas and check relevant essential goods supply and demand. We want to ensure the experience is smooth and feature rich for the users of the system to deliver the best results. Fortunately, in this instance this can be done through an internal development team and will not require interaction with external stakeholders. Simultaneously to this we will also be testing the integration of applying, approving and removing restrictions on essential items. Here we are simply testing if the restrictions imposed are communicated correctly to the ComCare backend infrastructure.

The third layer of testing will be the retail checkout system. For this phase we will test when a user’s card or NFC phone is scanned the communication flow is working as intended and the restrictions and purchase inventory are updated correctly. We will split this testing into two sections. Firstly, in house testing can be conducted with the POS system used by the supermarkets, this will ensure that any bugs or issues can be ironed out and corrected before involving the supermarkets. After this, similarly to the first phase of testing, we will then assign a development and UAT team to complete testing the process between the supermarket and ComCare infrastructure to ensure the process is working as required.

The next layer off testing will be that of the user web and application interface. The functional flow for the testing is as below:
Figure 9. Application flow chart.

Similarly, to the testing for the retail checkout system, we will be breaking down this section into two ensuring efficiency. Firstly, we will inhouse test as the development of the web interface and mobile application progresses. The progressive sections to test will include, user information communicating to ComCare infrastructure, restrictions to be communicated from ComCare infrastructure to the web interface and mobile applications, family restrictions to be updated along with current usage and finally any other features included. Once we have tested the feature set independently, the next step will be to internally test the entirety of the flow in house ensuring all features work in unison and the user interface is smooth and friendly.

Once we are confident internally the web interface and mobile applications are working as desired the next segment of testing will be to test with the retail checkout system, again we will take the same approach as phase one assigning a development and UAT team to complete this testing. Ensuring the applications are working as intended. For now, we currently won’t be getting public users involved however testing with internal teams to work through any issues before we proceed with public user testing.

Finally, the next step will be to test the system in its entirety from A – Z. This will be quite an extensive block of testing and will be broken up into two sections again. Firstly, using internal teams to go through every scenario ensuring they are working. This will give the team time to work on any issues before we get the public involved. The next step of testing will be to have public user testing, running through every scenario with the public. The aim would be by this point to be minimal issues thus giving the public a good representation of what the system is capable off and ensuring a smooth deployment when the time comes.
